Section 01

Abstract

HyperPress is an adaptive meta-compression engine that achieves 92.2% average compression across 8 diverse datasets, outperforming zlib-6 (84.7%) by 49% in size reduction and LZMA (89.7%) by 24%. Rather than implementing a novel compression algorithm, HyperPress introduces an intelligence layer that profiles each data block, races 7 preprocessing transforms against 5 compression backends, and selects the per-block winner based on empirical output size. The result is a self-contained HPRX binary format with CRC32 integrity verification.

Section 02

Motivation & Problem Statement

Traditional compression algorithms apply a single strategy to all data. zlib (DEFLATE) excels at text with its LZ77+Huffman approach. LZMA uses large dictionaries and range coding, ideal for structured binaries. bzip2 leverages BWT for redundant data. Yet no single algorithm is universally optimal.

Real-world data is heterogeneous. A single file may contain:

  • Text regions with high character redundancy
  • Binary headers with structured byte patterns
  • Sensor readings with strong delta correlation
  • Sparse sections dominated by null bytes
  • High-entropy regions that are incompressible
Core insight: Instead of choosing one algorithm, let multiple algorithms race on preprocessed variants of each block and keep the smallest output. The meta-compression overhead (14 bytes per block header) is negligible compared to the gains from per-block optimization.

Stage Details

  1. Block Splitting: Input data is divided into blocks of 4 KB to 128 KB. Block boundaries are guided by entropy variance to avoid splitting homogeneous regions.
  2. Block Profiling: Each block is analyzed for 6 features: Shannon entropy, delta-entropy, null-ratio, text-ratio, structure-score, and unique-byte-count.
  3. Preprocessing: Based on the profile, 3 to 7 candidate transforms are selected and applied to the block data.
  4. Backend Racing: Each preprocessed variant is compressed through up to 5 backends. The smallest output wins.
  5. Output Assembly: Winning blocks are serialized into the HPRX format with per-block CRC32 checksums.
Section 04

Block Analysis Engine

Each block is profiled using 6 numerical features that guide preprocessing selection:

Feature 1

Shannon Entropy

Measures information density. Range: 0.0 (constant) to 8.0 (random). Low entropy indicates high compressibility.

Feature 2

Delta Entropy

Entropy of byte-to-byte differences. Low delta-entropy indicates sequential correlation (sensor data, audio).

Feature 3

Null Ratio

Fraction of zero bytes. High null-ratio suggests sparse data ideal for RLE preprocessing.

Feature 4

Text Ratio

Fraction of printable ASCII bytes. High values indicate text data suited for BWT+MTF.

Feature 5

Structure Score

Detects repeating byte patterns at fixed intervals, indicating structured records (CSV, fixed-width).

Feature 6

Unique Bytes

Count of distinct byte values (0–256). Low unique count enables efficient Huffman/arithmetic coding.

H(X) = -∑i=0255 p(xi) · log2(p(xi))

HΔ(X) = H(xi - xi-1 mod 256)    for i = 1..n

Transform 1: Delta Encoding

Delta encoding stores the difference between consecutive bytes. For data with sequential correlation (sensor readings, audio samples), the delta stream has significantly lower entropy than the original.

d[i] = (x[i] - x[i-1]) mod 256    for i = 1..n,   d[0] = x[0]
Equation 3 — Delta encoding formula
def delta_enc(data):
    out = bytearray(len(data))
    out[0] = data[0]
    for i in range(1, len(data)):
        out[i] = (data[i] - data[i-1]) & 0xFF
    return bytes(out)

Transform 3: BWT + MTF

The Burrows-Wheeler Transform sorts all rotations of the input, grouping similar contexts together. Move-to-Front then converts the BWT output into a stream dominated by small integers, which compresses exceptionally well with entropy coders.

Transform 6: Nibble Split

Nibble splitting separates each byte into its high and low 4-bit nibbles, stored in two contiguous halves. This reduces entropy when high and low nibbles have different distributions, which is common in encoded data formats.

high[i] = x[i] >> 4,    low[i] = x[i] & 0x0F     output = high || low

The chain backend first compresses with zlib-9, then compresses the zlib output with LZMA. This two-pass approach can extract additional redundancy that neither backend finds alone — the DEFLATE pass reorganizes data in ways that benefit LZMA's dictionary matching.

Section 07

Racing Algorithm

The racing engine is the core innovation. For each block, it generates candidate combinations of (preprocessing, backend) and empirically selects the smallest output:

def race_block(block, profile):
    # Generate candidates based on block profile
    candidates = select_candidates(profile)

    best_size = len(block) + 1   # worse than stored
    best_result = None

    for (preproc, backend) in candidates:
        preprocessed = apply_transform(block, preproc)
        compressed = apply_backend(preprocessed, backend)

        if len(compressed) < best_size:
            best_size = len(compressed)
            best_result = (preproc, backend, compressed)

    return best_result

Candidate Selection Strategy

Not all 35 (7 transforms × 5 backends) combinations are tried for every block. The block profile guides selection:

  • Low delta-entropy: Delta and Delta-16 transforms are prioritized
  • High text-ratio: BWT+MTF is prioritized
  • High null-ratio: RLE is prioritized
  • High structure-score: Transpose is prioritized
  • High entropy (>7.5): Only Identity + Stored is tried (incompressible)

Typically 3 to 15 candidates are evaluated per block, keeping compression time practical.

Overhead analysis: For a 1 MB file split into 16 blocks of 64 KB each, the total overhead is 16 + (16 × 14) = 240 bytes — just 0.023% of the original data. This negligible overhead is the price paid for per-block optimization that saves kilobytes or megabytes.

Per-Dataset Win Analysis

Out of 8 datasets, HyperPress achieves the best compression on 4 (Sensor Data, Sparse Binary, Pattern+Noise, German Text). On the remaining 4, it places within 1–3% of the best individual compressor. Critically, on Sensor Data, HyperPress achieves 62.8% compression where zlib manages only 26.7% — a 2.35× improvement — thanks to Delta preprocessing exposing sequential correlation that DEFLATE cannot exploit alone.

Section 11

Conclusions

HyperPress demonstrates that meta-compression — the strategy of racing multiple preprocessing transforms and compression backends per data block — consistently outperforms any single static compressor across heterogeneous workloads.

Key findings:
  • 92.2% average compression ratio across 8 diverse datasets
  • 49% better size reduction than zlib-6, 24% better than LZMA
  • Per-block optimization adds only 14 bytes overhead per block
  • Delta preprocessing yields 2.35× improvement on sensor data
  • Zero external dependencies — Python 3.8+ stdlib only
  • Self-contained HPRX format with CRC32 integrity per block

Future Work

  • Parallel racing: Multiprocessing to compress candidate blocks concurrently
  • ML-guided selection: Train a small model on block profiles to predict optimal (preproc, backend) without full racing
  • Streaming mode: Enable compression of unbounded streams with adaptive block sizing
  • Additional transforms: Wavelet preprocessing for audio/image data, dictionary coding for repeated structures
